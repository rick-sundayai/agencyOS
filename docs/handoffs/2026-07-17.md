# Handoff: AgencyOS — 2026-07-17

## Context

AgencyOS is a Next.js + Postgres (Drizzle) ATS/CRM with an n8n-based agent runtime (Plans 1a-1c, already shipped). This session implemented **Phase 1d Part A**: a one-way migration of the book of business out of JobDiva into AgencyOS (candidates, resumes, job orders in; JobDiva goes read-only after cutover). Part B of the same plan (AWS infra: Terraform, RDS, n8n on ECS Fargate, Amplify, SES) was explicitly excluded from this session's scope, pending a separate Architect review of whether that AWS shape is the right one to commit to.

Implementation used `superpowers:subagent-driven-development`: one implementer subagent per task, one task-scoped reviewer per task (with fix rounds where findings surfaced), then one final whole-branch review — all run in an isolated git worktree, merged back to `main` only after everything was green.

## Key decisions

1. **Only Part A in scope now.** Part B needs Architect review of the AWS shape (Terraform + single-AZ RDS + ECS Fargate + Amplify, ~$60-90/mo) before any infra work starts. This is recorded as a scope note at the top of the plan file itself.
2. **Candidate dedupe fixed to use `jobdiva_id` as a first-class match key.** `src/services/ingest.ts`'s `ingestCandidate` (originally from Plan 1c) now checks `jobdiva_id` before falling back to email/phone — closes a bug where a candidate's email+phone both changing between JobDiva syncs (with a stable `jobdiva_id`) would silently create a duplicate row. This is an additive, backward-compatible schema/logic change.
3. **Client dedupe gap deliberately deferred, not fixed.** `clients` are still matched by `(org_id, name)` string only, despite a `jobdiva_id` unique index existing from Task 1 — because JobDiva's job BI row may not even expose a stable company/client ID. Unverifiable until someone runs live fixture capture. `docs/MIGRATION.md` now explicitly caveats this.
4. **Email validation regex anchored.** `mapCandidate`'s email guard was originally unanchored (substring match), which could let junk-wrapped emails through and defeat its own purpose. Fixed to `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`.
5. **Migration report output is now gitignored** (`docs/migration-report_*.md`) — it contains real candidate PII once run against production data, same treatment as `scripts/migration/fixtures/`.
6. **No `gh` CLI available in this environment** — team-summary/PR-issue creation was skipped by user request (solo dev).
